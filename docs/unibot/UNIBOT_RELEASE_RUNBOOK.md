# UniBot Release And Contributor Runbook

Status: public-safe draft runbook, not exam release.

The release runbook is generated by `unibot.release_runbook` and exposed
through:

- `POST /api/unibot/release-runbook`
- `POST /api/unibot/release-runbook-markdown`

## Purpose

This runbook gives future contributors a safe first path into the separate
UniBot track. It explains how to start the local service, load the browser
mantle, run the practice demo, validate demo feedback, triage work items, and
check public-draft readiness.

## Candidate Integrity Audit

After creating a public-draft candidate, run the read-only audit:

```text
unibot release audit ./unibot-review-candidate
```

It verifies the release manifest, recorded file sizes and SHA-256 hashes,
public-safety statuses, `not_cleared` exam status, zero provider calls, and the
source commit when the repository is available. A tampered or incomplete
candidate is blocked. The audit writes no files, calls no network or provider,
and cannot publish, merge, or grant institutional approval.

## Hash-Bound Verification Evidence

The candidate audit checks the handoff artifacts. The separate evidence step
binds actual local gate results to the same clean source commit:

```text
unibot release evidence --output ../unibot-release-evidence.json --repo .
```

It runs the fixed autonomy-preflight, synthetic Three Golden Rules, Ruff, mypy,
pip-audit, Python, browser, extension-package, Chrome-canary, pipeline,
public-safety, Guardian-benchmark, and source-card gates. The 3GR gate checks
that a bounded change has a reusable rule, positive and negative harness
evidence, and a human-gated improvement monitor. It stores only gate status, safe aggregate metrics,
duration, and SHA-256 hashes of command output;
raw logs, notebook text, learner attempts, credentials, and local paths are not
stored. The evidence file must remain outside the checkout so that adding it
cannot make the measured source tree dirty. A changed commit, dirty worktree,
missing gate, or modified evidence hash is blocked.

The direct public self-check is also available for a quick review without
running the full suite:

```text
unibot evaluate 3gr --json
```

This command uses only a fixed synthetic contract. It proves the release
boundary, not learning effectiveness, accessibility conformance, legal
compliance, university approval, or examination clearance.

## Human Release Handoff

The local branch becomes a public draft only through one deliberate human
GitHub action. The sequence is intentionally short:

1. Confirm that the working tree is clean and run the public-safety scan,
   Python suite, browser suite, branded Chrome Native-Messaging canary, the
   metadata-only public Colab canary with `npm run test:colab-canary`, and the
   local synthetic JupyterLab canary with `npm run test:jupyter-canary`. This
   canary starts and deletes a practice session, requests one real local A2
   tutor turn, and reads no notebook output or executes notebook code.
2. Build `unibot release candidate --output ./unibot-review-candidate` and
   run `unibot release audit ./unibot-review-candidate`. Keep the printed source
   commit and manifest hashes with the review record.
3. Run `unibot release evidence --output ../unibot-release-evidence.json --repo .`.
   Continue only when all fixed gates are green. The evidence is technical
   reproducibility, not institutional approval or exam clearance.
4. Generate the public-safe PR text from the audited candidate with
   `unibot release pr-draft --candidate ./unibot-review-candidate --output ./UNIBOT-PR-DRAFT.md --evidence ../unibot-release-evidence.json`.
   The command refuses stale, dirty, tampered, or unsafe candidates and does
   not contact GitHub. When the public Colab canary has produced a
   metadata-only JSON receipts, bind them to the same source commit in the
   final handoff with:
   `unibot release handoff --output ./unibot-release-handoff --evidence ../unibot-release-evidence.json --colab-canary ../colab-live-canary.json --jupyter-canary ../jupyter-live-canary.json`.
   The handoff rejects a stale, writable, symlinked, or content-bearing canary.
5. A human opens one **draft PR** from the Gretel branch to `main`. The PR
   must state: Gretel/Codex implementation and documentation, GLM provider
   calls and cost (`0` while parked), tests, public-safety result, remaining
   uncertainty, and the human merge decision still required.
6. GitHub CI repeats the required checks. A failed check blocks the PR; no bot
   merges, changes `main`, changes branch protection, or publishes a release.
7. Julius reviews the final diff after the last bot push and is the only person
   who may merge or release it.

This establishes a public review point, not institutional approval. The local
tutor remains usable without GLM and exam deployment remains `not_cleared`.

## Institutional Meeting Handoff

Before sending or presenting anything, generate the local public-safe packet:

```text
unibot institution bundle --output ./unibot-institution-review
```

The packet contains the RegulatoryProfile, human clearance board,
institutional presentation, `AccessibilityReviewV1` with its eight human-gated
checks, a plain-language `REVIEW-START-HERE.md`, a blank
`InstitutionalReviewDecisionTemplateV1`, source-card references, evidence hashes, and the explicit
Gretel/GLM/Julius authorship roles. It also carries the public synthetic
notebook, the metadata-only `PUBLIC-DEMO.md` script, and the deterministic
MV3 package for the same local demo. Demonstrate only that public synthetic
notebook. Do not add learner work, health or accommodation records, private
course files, local paths, credentials, or real examination tasks.
The reproducible public demo file is
`fixtures/public/synthetic_python_practice.ipynb`; use no other notebook for
the public meeting unless it has been separately reviewed.

The decision template is intentionally blank. It accepts only controlled
outcome IDs, role IDs, condition/question IDs, dates, and hashes. It does not
store meeting minutes and cannot grant exam, accessibility, privacy, legal, or
institutional approval.

For the live public walkthrough, run `unibot demo --markdown`. This uses only
`fixtures/public/synthetic_python_practice.ipynb`, sends no provider request,
executes no notebook code, and exercises the real deterministic tutor through
A0 to A4 in memory; A3 remains formula structure and A4 an incomplete scaffold.
Show the resulting safety boundary and ask the
institutional reviewers about teaching, inclusion, privacy, and the separate
exam track.

For a single local handoff, use the evidence produced above:

```text
unibot release handoff --output ./unibot-release-handoff --evidence ../unibot-release-evidence.json
```

This atomically builds the candidate, runs the read-only audit, writes the
human PR draft, copies the hash-only `RELEASE-EVIDENCE.json`, binds any Colab
and local synthetic JupyterLab canary receipts, and records their hashes in
`HANDOFF-MANIFEST.json`. A failed step leaves no partial handoff and does not
contact GitHub. The JupyterLab receipt proves high-confidence cell metadata
capture plus one bounded local tutor turn in a synthetic notebook; it proves no
learning effect, institutional approval, or examination clearance.

The meeting asks five bounded questions: who owns the institutional decision;
whether the local practice purpose fits the named module; which accessibility
and assistance functions are appropriate; which data retention and deletion
rules apply; and which separate written decision would be required for any
future exam track. The meeting records conditions and open questions, not an
automatic approval.

## Hard Boundary

- Public draft means review and collaboration, not exam clearance.
- UniBot is not a proctoring system.
- UniBot is not KI-detection evidence.
- UniBot is not an automatic grader.
- UniBot does not decide accommodation or official support status.
- Contributors must not add private course files, emails, local paths, personal
  health details, real exam notebooks, grades, or copied private classroom
  artifacts to public files.

## Public Quickstart

1. Start the local UniBot service with `python3 -m unibot.server`.
2. Check `GET /api/unibot/health`.
3. Load `unibot/browser_extension` as an unpacked extension in a
   Chrome-compatible browser.
4. Optional: run the separate track smoke `python3 scripts/unibot_pipeline_smoke.py`
   (or `--json` / `--run-tests` for deeper validation).
5. Run `POST /api/unibot/demo-run`.
6. Run `POST /api/unibot/readiness-check`.
7. Validate feedback with `POST /api/unibot/demo-feedback/validate`.
8. Build triage with `POST /api/unibot/demo-feedback/triage`.
9. Draft reviewable GitHub items with `POST /api/unibot/github-issue-bundle`.
10. Build the internal review board draft with `POST /api/unibot/review-board-packet`.
11. Check PM status with `POST /api/unibot/orchestration/command-center`.
12. Validate parallel-chat handoffs with `POST /api/unibot/orchestration/handoff/validate`.
13. Run `POST /api/unibot/completion-audit` before claiming project closure.
    The completion audit carries a workspace-card closure alignment that links
    public-draft completion status, readiness snapshot evidence,
    command-center route status, and the not-cleared deployment line through
    hashes only. It is a technical public-draft closure aid, not public release,
    provider-call approval, grading, proctoring, KI-detection evidence, or exam
    clearance.
14. Inspect course extraction work with `POST /api/unibot/course/extraction-queue`.
15. Prepare and validate the rights/privacy gate with
    `POST /api/unibot/course/extraction-decision-packet` and
    `POST /api/unibot/course/extraction-decision/validate`.
16. Inspect or store the local rights/privacy decision through
    `POST /api/unibot/course/extraction-decision/local-intake` and
    `POST /api/unibot/course/extraction-decision/local-intake/record`. The
    intake packet shows the minimum decision template, hash-only journal gate,
    OCR-first readiness, receipt-journal status, and post-run report status
    without returning raw decisions, raw course text, or local paths.
17. Prepare the local Decision-Record workspace with
    `POST /api/unibot/course/extraction-decision/workspace/prepare`; it writes
    a JSON template and hash-only workspace manifest under the local UniBot
    guardian directory and returns a dry-run receipt only. When the written
    record is ready, use
    `POST /api/unibot/course/extraction-decision/workspace/record` to append the
    valid decision as hash-only journal evidence. Neither route starts OCR or
    returns raw decisions, raw course text, or local paths.
18. Run the controlled OCR-first Batch-1 operator with
    `POST /api/unibot/course/ocr-first/operator-run`. It first checks the
    current workspace dry-run receipt and only starts private OCR when the
    local decision journal is valid and `operator_confirmed_dry_run` is true.
    After running it writes hash-only receipts and returns Progress,
    Manifest-Update, and Tutor-Coverage summaries from the receipt journal.
19. Inspect the gated local extraction operator with
    `POST /api/unibot/course/extraction-operator-packet`.
20. Build the extraction batch plan with
    `POST /api/unibot/course/extraction-batch-plan`; it is a plan-only
    receipt backlog, not OCR/transcription execution. Use
    `job_types: ["ocr"]` for OCR-first batches when video transcription is
    waiting on sidecars or ASR.
21. Prepare the selected batch receipt packet with
    `POST /api/unibot/course/extraction-batch-receipt-packet`; it contains
    receipt templates and the human-review checklist, not raw text, and accepts
    the same `job_types` filter.
22. Run the private text-container extraction harness with
    `POST /api/unibot/course/private-extraction/run-batch` only after a valid
    local extraction decision. It currently supports DOCX, text-PDF, and PPTX,
    writes raw text only to private local artifacts, and stores hash-only
    receipts. Use `job_types: ["ocr"]` plus `max_jobs: 12` for the OCR-first
    Batch 1 path. When the written local-extraction decision has already been
    recorded in the hash-only decision-record journal, pass
    `decision_record_journal_path` instead of resending the raw decision record;
    the runner uses the accepted journal hash as the gate context and still
    returns no raw decision text or local paths.
23. Run the private video transcription harness with
    `POST /api/unibot/course/video-transcription/run-batch` only after a valid
    local extraction decision. It ingests sidecar transcripts when present,
    reports missing local ASR adapters clearly, and returns only hash/count
    receipt metadata.
24. Validate extraction receipts with
    `POST /api/unibot/course/extraction-receipt/validate`; receipts contain
    hashes, counts, and review state only, not raw text.
25. Store accepted extraction receipts with
    `POST /api/unibot/course/extraction-receipts/append`, then inspect
    `POST /api/unibot/course/extraction-receipts/summary`; the journal is
    local-only and hash-only.
26. Record human review decisions and prepare the private apply plan with
    `POST /api/unibot/course/extraction-review/validate` and
    `POST /api/unibot/course/extraction-review/apply-plan`; accepted decisions
    append `reviewed_for_private_tutor` receipts plus hash-only review records,
    then return private Manifest-Update and Tutor-Coverage previews without
    writing manifests, starting tutor indexing, returning raw text, or exposing
    local paths.
27. Preview and optionally apply private manifest metadata with
    `POST /api/unibot/course/extraction-manifest/apply-dry-run`; the default
    dry-run returns a metadata delta plus ExamScopeMap/Tutor-Coverage previews.
    It writes a local private manifest only when
    `operator_confirmed_manifest_apply` is true, and still does not start tutor
    indexing or clear exam deployment.
28. Preview and optionally build the private tutor source-anchor index with
    `POST /api/unibot/course/tutor-index/dry-run`; the default dry-run reads
    the private manifest and returns hash-only retrieval anchors, Source-Card
    coverage, and ExamScopeMap gaps. It writes a local index only when
    `operator_confirmed_tutor_index_build` is true.
29. Preview an A0-A2 tutor answer from the hash-only index with
    `POST /api/unibot/course/tutor-index/respond-dry-run`; it returns only
    source-anchor metadata, Socratic questions, and a Help-Ledger preview, and
    keeps the raw query hash-only.
30. Run the connected private tutor-use flow with
    `POST /api/unibot/course/private-tutor-use-flow/dry-run`; it chains private
    Manifest-Apply, Tutor-Index build, index-guided A0-A2 answer,
    optional Help-Ledger append, and Study-Receipt validation. Default remains
    dry-run; writes require explicit operator confirmation.
31. Run the controlled exam workspace with
    `POST /api/unibot/exam-workspace/run-dry-run`; it links material freeze,
    notebook cell checkpoint, private tutor sidecar, Help-Ledger evidence,
    Study-Receipt validation, and export receipt. Default remains dry-run;
    local writes require explicit operator confirmations and the receipt stays
    `not_cleared`.
32. Adapt a local notebook checkpoint with
    `POST /api/unibot/exam-workspace/notebook-checkpoint/adapt`; it turns a
    locally captured cell or hash-only checkpoint into hashes, Cell-Checkpoint
    metadata, Study-Receipt validation, and Help-Ledger preview without
    returning raw cell text, notebook code, values, local paths, final
    interpretation, grading, proctoring, AI detection, or clearance claims.
33. Review the Python exam skill drilldown with
    `POST /api/unibot/course/exam-skill-drilldown`; it turns material coverage
    into a selectable skill map with selected-skill readiness, remaining
    OCR/video/source gaps, notebook-checkpoint template, Help-Ledger preview
    template, and a dry-run operator action template while staying metadata-only
    and `not_cleared`. In the side panel, `Skill-Workspace Dry-Run starten`
    uses this template to run the selected skill through the Start Exam
    Workspace operator flow without enabling local writes by default.
34. Review the Start Exam Workspace operator receipt with
    `POST /api/unibot/exam-workspace/operator-run`; it merges material
    coverage, notebook checkpoint, A2 tutor state, Help-Ledger preview, export
    receipt, and individual operator confirmations into one human-readable
    dry-run view. Default remains no local writes and `not_cleared`.
35. Review the Exam Workspace Session Console with
    `POST /api/unibot/exam-workspace/session-console`; it keeps selected skill,
    workspace status, notebook checkpoint hash, A0-A2 tutor state, Help-Ledger
    preview, export receipt, open confirmations, and repeat dry-run evidence in
    one compact work surface without exposing raw queries, course raw text,
    notebook code, local paths, grading, proctoring, AI detection, or clearance
    claims.
36. Review the run history export package with
    `POST /api/unibot/exam-workspace/run-history-export-review`; it aggregates
    Session Console receipts into selected skills, checkpoint hashes,
    help-level profile, Help-Ledger preview metadata, blockers,
    operator-confirmation profile, export receipt, and reflection status without
    exposing raw work or claiming exam clearance.
26. Validate an intentional extraction deferral, if the current scope should
    proceed without finishing every remaining OCR/transcription job, with
    `POST /api/unibot/course/extraction-deferral/validate`.
27. Build the extraction completion report with
    `POST /api/unibot/course/extraction-completion-report`; it classifies jobs
    as complete only through no queue, reviewed receipts, failed/skipped
    receipts, or valid deferral evidence.
28. Build the extraction progress report with
    `POST /api/unibot/course/extraction-progress-report`; use it to create
    human review queue items and private manifest-update candidates. It can
    consume the local receipt journal through `receipt_journal_path`.
29. Build the extraction manifest update plan with
    `POST /api/unibot/course/extraction-manifest-update-plan`; it converts
    reviewed receipts into private tutor metadata candidates without writing
    manifest files.
30. Build the tutor coverage plan with
    `POST /api/unibot/course/tutor-coverage-plan`; it forecasts current versus
    projected skill readiness before rebuilding ExamScopeMap.
31. Run the material coverage control report with
    `POST /api/unibot/course/material-coverage/run`; it condenses private
    Manifest/Receipt/Index evidence, OCR/video gaps, notebook exercise
    readiness, and the next A0-A2 Exam-Workspace start point without raw text,
    notebook code, local paths, or clearance claims.
32. Launch that selected start point with
    `POST /api/unibot/exam-workspace/launch-flow/dry-run`; it prepares the A2
    query template, notebook cell checkpoint, source-anchor hint, Help-Ledger
    preview, and export receipt. Default remains dry-run; local writes require
    explicit operator confirmations and the receipt stays `not_cleared`.
31. Build the study session plan with
    `POST /api/unibot/course/study-session-plan`; it turns reviewed tutor
    anchors into retrieval, notebook, reflection, and spacing tasks.
32. Validate study session receipts with
    `POST /api/unibot/course/study-session-receipt/validate`; each receipt
    records prediction, retrieval response, source anchor, smallest notebook
    action, help level, and reflection as hashes/flags only.
33. Build the study session review report with
    `POST /api/unibot/course/study-session-review-report`; use it as
    human-reviewable learning evidence, not as grading, AI detection, ranking,
    or an Eigenleistung percentage.
34. Inspect the stakeholder clearance board with
    `POST /api/unibot/institutional-clearance/board`.
35. Validate any written scope record with
    `POST /api/unibot/institutional-clearance/validate`; validation is a shape
    check and does not itself create real exam clearance.
36. Build the stakeholder submission bundle with
    `POST /api/unibot/stakeholder/submission-bundle`; manually review before
    using any message draft.
34. Prepare one lane-specific manual decision request with
    `POST /api/unibot/stakeholder/decision-request`, then generate the review
    text with `POST /api/unibot/stakeholder/decision-request-markdown`; UniBot
    does not send it and the packet is not approval.
35. Validate a manual request receipt with
    `POST /api/unibot/stakeholder/decision-request/validate-receipt`; receipt
    validation stores a submission-reference hash only and changes no gate.
36. Record prepared request and validated receipt evidence in the local journal
    with `POST /api/unibot/stakeholder/decision-journal/append` or
    `POST /api/unibot/stakeholder/decision-journal/append-prepared-request`;
    inspect it with `POST /api/unibot/stakeholder/decision-journal/summary`.
37. Validate incoming written responses with
    `POST /api/unibot/stakeholder/decision-state`; the result stores hashes and
    gate state only, not raw decision text.
38. Store accepted written decision records locally with
    `POST /api/unibot/stakeholder/decision-record-journal/append`, then inspect
    `POST /api/unibot/stakeholder/decision-record-journal/summary`; the journal
    stores hashes and scope metadata only and keeps exam deployment
    `not_cleared`.

## Contributor Rules

- Use synthetic tasks or public source cards only.
- Keep course-material work staged until permission, extraction status, review
  status, skill tags, and public policy are explicit.
- Add tests for blockers, privacy handling, and exam-boundary behavior.
- Treat the browser overlay as practice evidence only.
- Review generated issue drafts manually before public posting.
- Keep public language precise: "public draft", "practice demo", and "not
  officially cleared for exam use".

## Release Gates

- Public-safety scan passes over UniBot docs, code, extension files, and tests.
- Red-team smoke passes for solution leakage, privacy leakage, prompt
  injection, source risk, and exam boundary.
- Publication package is ready as a public draft only.
- Readiness check is green while `exam_deployment_status` remains
  `not_cleared`.
- GitHub issue bundle is manually reviewed before publication.
- Review board packet should be reviewed with institutional stakeholders before exam-control planning.
- Any future exam variant needs written review by the responsible university
  authorities before use.

## Troubleshooting

- If the local API is offline, start the local service and retry the health
  endpoint.
- If the practice gateway reports that its port is in use, stop the other local
  Jupyter service or start again so the Companion can choose a free loopback
  port. A failed Jupyter start is reported as blocked and does not create a
  misleading active gateway record.
- If the side panel cannot access selected page text, use the visible
  copy/paste fallback.
- If feedback is blocked, remove private details and validate again.
- If readiness is blocked, inspect the failed check and fix that gate first.
- If multiple chats are working in parallel, use the Command Center section in
  the browser side panel before handing work back.
- If the completion audit returns `goal_complete: false`, treat the listed open
  technical items as the next project-manager backlog instead of claiming
  closure. Real-world exam clearance appears as a reminder and does not block
  technical bot delivery.
- If extraction jobs are blocked, record a rights/privacy decision before
  running local OCR or transcription.
- If a decision validates, it authorizes only local private extraction and
  human review before tutor indexing; it does not authorize exam deployment or
  public raw text release.
- If the operator packet is blocked, provide a valid local-extraction decision
  record before any OCR/transcription run. If a receipt is blocked, remove raw
  text fields, fix the decision hash, or complete required job metadata.
- If the extraction batch plan is ready, it is still only a work queue. Run
  local private OCR/transcription outside public output and validate one receipt
  per job before claiming progress.
- If the private extraction runner is blocked, provide a valid local
  rights/privacy decision record first. It supports DOCX, text-PDF, and PPTX
  containers only and does not replace image-only PDF OCR or video
  transcription.
- If the progress report is waiting for receipts, run the local private jobs
  after the validated decision. If it has review candidates, human-review them
  before updating the private tutor manifest.
- If validated extraction receipts do not appear in later reports, append them
  to the local receipt journal and pass `receipt_journal_path` to progress,
  batch, manifest, coverage, or study APIs.
- If receipts are ready but manifest candidates are missing, run the human
  review apply-plan with hash-only review decisions. It can mark accepted
  receipts `reviewed_for_private_tutor`, but it still does not write manifests
  or start tutor indexing.
- If private manifest candidates are ready but the tutor still has no new
  anchors, run the private manifest apply dry-run, inspect the delta and
  ExamScopeMap preview, then rerun with explicit operator confirmation only if
  the local metadata manifest should be updated. Tutor indexing is still a
  later explicit step.
- If extraction completion remains open, finish reviewed/failed/skipped
  receipts for every queued job or validate a role-reviewed deferral record for
  the current scope. Deferral does not publish, index, grade, or clear exam use.
- If a study-session receipt is blocked, add missing prediction, retrieval
  response, smallest notebook action, source anchor, and reflection. Repeat the
  task if A6 or final-solution exposure occurred.
- If the study-session review report is partial, validate one hash-only receipt
  per planned task and keep the result formative, non-grading, and free of raw
  student text.
- If a clearance record validates, it proves the record matches the required
  schema for that scope only; keep real exam deployment `not_cleared` until the
  responsible people have issued written clearance and the deployment gate is
  intentionally changed. Missing clearance is a real-world reminder, not a
  technical completion blocker.
- If the stakeholder submission bundle is ready, it still means only “ready for
  human review.” It does not send anything, authorize extraction, or clear exam
  deployment.
- If a stakeholder decision request is ready, manually review it before any
  real communication. A receipt proves only that a request was handled; it is
  not the rights/privacy decision or exam clearance itself.
- If the stakeholder decision journal has entries, treat them as continuity
  evidence only. They do not authorize OCR/transcription, exam use, or any
  deployment switch.
- If the decision-state packet shows valid records, use it as gate evidence.
  Local extraction still needs receipts and human review; exam deployment still
  needs an explicit manual switch.
- If the decision-record journal has accepted entries but nothing deploys, that
  is correct. The journal is human-review evidence for completion audit, not a
  sending tool or deployment switch.
